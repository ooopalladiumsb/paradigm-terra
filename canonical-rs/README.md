# `paradigm-terra-canonical` (Rust parity)

Rust parity implementation of the **Paradigm Terra Canonical Encoding
Specification v1.3 (SCF)** with the v0.10.0-draft domain-tag extensions
(`STATE_ROOT_V1`, `DSL_V1.2`). It mirrors the TypeScript reference
(`../canonical`, `@paradigm-terra/canonical`) byte-for-byte.

## Parity status

The test `tests/parity.rs` loads the committed
`../canonical/vectors/golden.json` (generated by the TypeScript reference) and
independently recomputes every vector with this crate:

```
All 44 parity checks passed against TypeScript golden vectors.
```

44 field comparisons across all 17 golden vectors — integers, UTF-8 NFC, TON
address, restricted JCS, DSL v1.1/v1.2 hashes, CAL hash, stream-tree and
binary Merkle, genesis state root, framing, and the full domain-tag registry.

This satisfies the **Rust** half of the promotion gate. The vectors stay
`PRE-NORMATIVE` until the **Go** parity implementation also passes; only then
should `golden.json` be promoted to `NORMATIVE`.

## Building / testing

```sh
cargo test          # builds + runs the SHA-256 KAT and the parity suite
cargo clippy --all-targets
```

### Why a musl static cross-build

`.cargo/config.toml` pins the target to `x86_64-unknown-linux-musl` and links
with the toolchain's bundled `rust-lld`. This makes the crate build with **no
system C toolchain**: rustup ships the musl CRT (`crt1.o`, `libc.a`, …) under
the target's `self-contained/` directory, and `rust-lld` drives the link. The
resulting static binary runs unmodified on a glibc host.

For this to work the dependency tree must contain **no build scripts and no
proc-macros** (those would have to link for the gnu host). Hence:

- **SHA-256 is vendored** (`src/sha256.rs`) instead of using `sha2`, which pulls
  in `cpufeatures`/`typenum` build scripts. Correctness is pinned by the golden
  vectors and a NIST known-answer test. To switch to `sha2` in an environment
  with a C toolchain, drop the `pub use crate::sha256::sha256;` in `hash.rs` and
  add `sha2 = "0.10"`.
- **The parity test reads `golden.json` with this crate's own JCS parser** rather
  than `serde_json`, avoiding the `serde_derive` proc-macro.

Remaining dependencies — `hex` and `unicode-normalization` (+ `tinyvec`) — are
all pure (no build scripts, no proc-macros).

### Unicode version note

NFC tables come from `unicode-normalization`. The TS reference pins Unicode 15.1
(Node 22, ICU 73+). For full conformance the crate's Unicode version must match
the reference; all current NFC golden vectors (`U+0065 U+0301 → U+00E9`) are
stable across Unicode versions.
